Topic 1

互动计数 — 从零开始

0. 从一个帖子说起

帖子 123 被 3 个用户点过赞（5、99、203），被 1 个用户收藏（99）。

问题例子答案
"某个用户点过吗？"用户 99 点过赞吗？是
"总共有多少？"几个赞？3

只存数字 likeCount=3 → 回答不了"谁点过"。只存集合 {5,99,203} → 100 万点赞的 Set 占 30-50MB。

这个项目同时用两个结构：

# 位图 — "哪些用户点了赞" bm:like:knowpost:123:0 → bit[5]=1, bit[99]=1, bit[203]=1 # 20 字节二进制串 — "总共几个赞/收藏" cnt:v1:knowpost:123 → byte[4..7]=3(like), byte[8..11]=1(fav)
为什么查总数不直接用 BITCOUNT？ 百万点赞 ≈ 30 个分片 key。每次读都 BITCOUNT 30 次太贵。不如维护一个 20 字节的小结构，一次 GET 搞定。

代码里管位图叫"事实层"（source of truth），管二进制串叫"计数快照"（SDS）——就是一个固定格式的 byte 数组，没什么玄学。

1. 三个 Redis 结构 — 各司其职

1.1 位图 — 点赞/取消时写入

bm:like:knowpost:123:0 ← chunk=0, 存 userId 0~32767 bm:like:knowpost:123:1 ← chunk=1, 存 userId 32768~65535 # chunk 怎么算？ userId = 100000 → chunk = 100000/32768 = 3 → bit = 100000%32768 = 1696

每个 chunk 固定 32768 位 = 4KB。1000 万点赞不是 1000 万位的单 key，而是 ~305 个各 4KB 的小 key——避免大 key 导致 Redis 主从同步阻塞。

1.2 计数快照（SDS）— 读计数的入口

cnt:v1:knowpost:123 → 20 字节, 5 字段 × 4 字节大端 int32 byte 0-3: 0 (read 预留) byte 4-7: 3 (like count) byte 8-11: 1 (fav count) byte 12-15: 0 (comment 预留) byte 16-19: 0 (repost 预留)

设计决策 4 字节 int32（最大 21 亿）够用且省空间；大端序 = Java ByteBuffer 默认 + 网络字节序；固定长度 20 字节 → 长度校验就是完整性校验。

1.3 聚合桶（Redis Hash 缓冲）— 写计数的入口

agg:v1:knowpost:123 → Redis Hash field "1" → "5" (like 待处理增量 +5) field "2" → "-2" (fav 待处理增量 -2)

核心价值 Kafka 每秒几千个点赞事件。如果每个事件都直接改 SDS（Lua GET→解码→+1→编码→SET），写放大严重。 聚合桶先 HINCRBY 累加（单命令），每秒只折叠一次。1000 次 HINCRBY → 1 次 Lua 脚本。

比喻： 位图 = 考勤表（每人一格） · 计数快照 = 黑板上的数字 · 聚合桶 = 助教手里的便签
查某人是否出席 → 考勤表 · 查总人数 → 黑板 · 有人进出 → 先记便签，每分钟擦一次黑板 · 黑板数字坏了 → 重新数考勤表
2. 写入链路 — 从 HTTP 到 SDS
HTTP POST /api/v1/action/like {"entityType":"knowpost","entityId":"123"} │ ▼ Java CounterServiceImpl.like() → toggle() ├─ 1. chunk = uid/32768, bit = uid%32768 ├─ 2. Redis Lua TOGGLE_LUA: GETBIT→检查→SETBIT (原子) └─ 3. changed=1 → ① Kafka: CounterEvent ② Spring Event ▼ Kafka CounterAggregationConsumer.onMessage() ├─ 4. HINCRBY agg:v1:knowpost:123 "1" 1 └─ ack → 只有成功才提交位点 ▼ Redis @Scheduled(fixedDelay=1000) CounterAggregationConsumer.flush() ├─ 5. 扫描 agg:* key → INCR_FIELD_LUA: delta 折叠进 SDS │ → DECR_FIELD_LUA: 扣减聚合桶 field └─ Hash 变空 → DELETE agg key
面试必问：为什么 TOGGLE_LUA 要先 GETBIT 再 SETBIT？
SETBIT 返回旧值，但两个并发请求都 SETBIT offset=5 to 1：请求 A 返回 0（旧值）→ 产生事件；请求 B 返回 1（旧值，因 A 先执行）→ 也产生事件。两次都以为自己改变了状态！
正确做法：Lua 里先 GETBIT，已是目标状态就 return 0，不是才 SETBIT。判断 + 写入在 Redis 单线程内原子完成。
3. 读取链路 & 判存

getCounts() 单个读

GET cnt:v1:knowpost:123 → 20字节 正常: byte[4..7]解出like, [8..11]解出fav 异常: 长度≠20 → 触发重建

getCountsBatch() 批量读

Pipeline GET 20个SDS key 一次网络往返 缺失的返回 0，不触发重建
isLiked() 直接读位图，不依赖 SDS。 即使 SDS 挂了在重建，用户看到的点赞状态永远准——因为位图是事实源。
4. BITCOUNT 重建 — 三层保护机制

触发条件：SDS key 不存在，或长度 ≠ 20。原因：Redis 内存淘汰、运维误删、主从切换。

触发 getCounts()，SDS 缺失 │ ├─ 第0层 inBackoff() → 退避窗口内？直接返回0 │ ├─ 第1层 限流 Redisson RRateLimiter 3 permits/10s │ 拿不到 → escalateBackoff() → 返回0 │ ├─ 第2层 分布式锁 RLock.tryLock(0) 非阻塞 │ 拿不到 → escalateBackoff() → 返回0 │ # tryLock(0)：不等锁。别的线程在做，下次请求SDS已存在，直接读就行 │ ├─ 第3层 持锁重建 │ ① KEYS bm:like:knowpost:123:* → Pipeline BITCOUNT → sum │ ② SET cnt:v1:knowpost:123 = 新SDS │ ③ HDEL agg:v1:knowpost:123 "1" "2" ← 清聚合桶，防重复加算 │ ④ resetBackoff() │ └─ 第4层 指数退避 连续失败 → delay=500ms×2^exp, cap=30s, exp cap=10
机制解决的问题不解决的
限流同一实体重建频率过高，刷爆 Redis不同实体各自有限流器
分布式锁同一实体的并发重建竞争锁超时（watchdog 自动续）
指数退避反复失败时快速失败退避窗口内返回0，数据不准

先限流（粗筛）→ 再抢锁（精筛）→ 失败就退避（自保）

Topic 2

用户关系 — 双表镜像 + Outbox 事件驱动

0. 核心问题 & 数据模型

朴素方案：

INSERT INTO following (from_user_id, to_user_id) VALUES (1, 2); SELECT from_user_id FROM following WHERE to_user_id=2 ORDER BY created_at LIMIT 20;

问题是5000 万粉丝的大 V——即使走索引也要扫描 5000 万行排序。FB 式 fan-out 读扩散扛不住。

解决方案：同一个操作存两张表，行列互换

following (主表) follower (镜像) ┌──────────────────────┐ ┌──────────────────────┐ │ id: 1001 │ │ id: 2001 │ │ from_user_id: 1 │ │ to_user_id: 2 │ │ to_user_id: 2 │ │ from_user_id: 1 │ │ rel_status: 1 │ │ rel_status: 1 │ └──────────────────────┘ └──────────────────────┘ UK: (from, to) UK: (to, from) 查"我关注了谁" 查"谁关注了我"

就像同一份名单按不同字段排序存两份——查的时候不用翻页。

面试提醒： 代码里叫"一主多从"，但容易和 MySQL 主从复制混淆。面试时用"双表镜像"更安全。

为什么不用一张表 + 两个索引？ ① 读写隔离——粉丝列表 QPS 远高于关注列表，可分库；② 与 Outbox 异步边界一致；③ 每表只维护一个唯一索引，插入竞争小。

1. 写入链路 — 17 步全路径
POST /api/v1/relation/follow?toUserId=2 │ ▼ RelationServiceImpl.follow() ├─ 1. TOKEN_BUCKET_LUA: rl:follow:1, cap=100, rate=1/s ├─ 2. @Transactional 开始 ├─ 3. INSERT following (id=random, ON DUPLICATE KEY UPDATE) ├─ 4. INSERT outbox (同事务!) └─ 5. 事务提交 → binlog 记录两条 INSERT ▼ CanalKafkaBridge (SmartLifecycle, 独立线程) ├─ 6. getWithoutAck(batchSize) → 拉binlog ├─ 7. 过滤: table="outbox", type=INSERT ├─ 8. 提取 payload → {"table":"outbox","data":[{"payload":"..."}]} ├─ 9. kafka.send("canal-outbox") └─ 10. ack(batchId) → 推进位点 ▼ CanalOutboxConsumer → RelationEventProcessor.process() ├─ 11. SETNX "dedup:rel:FollowCreated:1:2:<id>", TTL=10min ├─ 12. INSERT follower (id, to_user_id, from_user_id, rel_status=1) ├─ 13. ZADD "uf:flws:1" {now} "2" + EXPIRE 2h ├─ 14. ZADD "uf:fans:2" {now} "1" + EXPIRE 2h ├─ 15. incrementFollowings(1, +1) → ucnt:1 └─ 16. incrementFollowers(2, +1) → ucnt:2
为什么 Outbox 和 following 必须同事务？
分开写 → DB 提交了但 Outbox 没写（MySQL 挂了）→ follower 表永不更新、ZSet 永不更新、计数永不 +1 → 数据永久不一致且无法自动恢复。
同事务 → 要么都成功（binlog 有记录 → Canal 能读到 → 完整链路），要么都失败（用户看到"关注失败"，可以重试）。
2. 设计深挖

Canal vs 直接发 Kafka

比喻： 直接发 Kafka = 做完事后自己打电话通知（可能忘打/打不通）。Outbox + Canal = 在公告板贴便条（binlog），Canal 专门负责抄公告。只要事务提交了，binlog 里就一定有，Canal 就一定能读到。
可靠性复杂度为什么这个系统用它
直接发 Kafka可能丢（网络问题）低计数系统——丢一条事件 = 赞数少1，下次重建修复
Outbox + Canal不丢高关系系统——丢一条 = 数据永久损坏

去重 SETNX 10 分钟的边界

边界1: 消息延迟 > 10min

Kafka rebalance 15min → SETNX key 已过期 → 重复处理。follower 表 ON DUPLICATE KEY 还行，但计数器会 +1 两次。下次 300s 校验修复。

边界2: 取关后重关注

第二次关注生成新 following id (B≠A) → 去重 key 不同 → 不误判。

令牌桶限流

capacity=100, rate=1/s。允许 1 秒内 burst 关注 100 人（批量导入），之后等 1 秒才能再关注。用 Lua 脚本 + Redis Hash 实现，redis.call('TIME')[1] 获取单调秒数。

3. 读取链路 & 计数器自愈
GET /api/v1/relation/followers?userId=2&limit=20&offset=0 │ ├─ L0: Caffeine 本地 Top 缓存 (仅大V, ≥50万粉, 前500名) │ ├─ L1: Redis ZSet "uf:fans:2", ZREVRANGE, TTL=2h │ score = created_at 毫秒戳 → 最近关注排前面 │ └─ L2: MySQL follower 表 → fillZSet → 回填 → EXPIRE 2h
自愈机制： 每 300 秒（SETNX 锁），SELECT COUNT(*) 对比 SDS 值。不一致 → rebuildAllCounters() 从 DB 重算全部 5 个字段。最多 300 秒自动发现并修复，不需人工介入。

Topic 3

Feed 流 — 多级缓存 + 热点探测

0. 三种场景，三种缓存策略
场景策略核心考量
公共 Feed片段缓存：ID列表 + Item JSON共享率极高。拆成 ids+items，item 粒度独立失效
我的发布整页 JSON用户独占。一页一个 key，省事
内容详情整页 JSON + null cache单条内容。额外 null cache 防穿透

为什么公共 Feed 要拆片段？ 如果不拆，每页一个 Redis key 存 20 条帖子的完整 JSON。点赞时需要一个一个页面找出来改计数——你不知道帖子 123 在前 100 页的哪些页面上。拆成 ids + items 后，点赞只需改一个 item key。

1. 公共 Feed 请求链路 — 三级缓存
GET /api/v1/knowpost/feed?page=1&size=20 (uid=99) │ ├─ L0 Caffeine (15s) → feedPublicCache.get(localPageKey) │ 命中 → recordHotKey → enrich(liked/faved实时) → 返回 │ ├─ L1 Redis片段 → assembleFromCache(idsKey, hasMoreKey) │ ① LRANGE idsKey 0 19 → 20个ID │ ② MGET feed:item:101 ... 120 → 20条JSON │ ③ counterService.getCounts() + isLiked() → 实时计数+用户状态 │ 命中 → put Caffeine → enrich → 返回 │ ├─ singleFlight │ synchronized(idsKey lock) { double-check → ... } │ └─ L2 MySQL SELECT ... LIMIT 21 OFFSET 0 → rows≥20? hasMore → writeCaches: LPUSH idsKey | SET item JSON | SADD 反向索引 → put Caffeine → enrich → singleFlight.remove(idsKey) → 返回
用户状态不入缓存： liked/faved 永远不在 Redis 缓存中。返回前通过 enrich() 实时查 CounterService.isLiked()。 缓存是共享的，用户状态是私有的——不能污染。
2. 设计深挖 — 小时槽 · 反向索引 · single-flight · 抖动

小时槽分片 hourSlot

缓存 key 嵌入 hourSlot = now / 3600000。新帖发布只影响当前小时的 ids key，上一小时的缓存靠 TTL 自然过期。避免"写一篇文章让 100 页缓存全失效"的雪崩。

反向索引 — 点赞时精准更新

监听 CounterEvent (like/fav 变化) ├─ SMEMBERS feed:public:index:{eid}:{hourSlot} ← 当前小时 ├─ SMEMBERS feed:public:index:{eid}:{hourSlot-1} ← 上一小时 ├─ 对每个页面 key: 更新 Caffeine (保留 liked/faved) + 更新 Redis (清除 liked/faved) └─ 写回Redis时保留原TTL (writePageJsonKeepingTtl)

single-flight

锁 key 用 idsKey（Redis 层面）而不是 localPageKey（本地）。跨实例全局唯一。只有持锁线程回源 DB，其他线程 double-check 等缓存填满。注意 finally { singleFlight.remove(idsKey); } 防泄漏。

随机抖动

缓存层TTL浮动%
item/id 片段60s + rand(0~30)50%
hasMore 软缓存10s + rand(0~10)100%
我的发布30s + rand(0~20)67%
Caffeine 本地固定（无抖动）—

Caffeine 是单实例，不存在"集群 100 个节点同时过期同一个 key"。雪崩只发生在共享 Redis 层。

3. HotKeyDetector — 滑动窗口数值走读

60 秒窗口，6 段 × 10 秒。每个 key 一个 int[6]。

帖子 #888 的访问时间线： 第 0-10s: 100次 → seg[0]=100 第10-20s: 150次 → seg[1]=150 第20-30s: 200次 → seg[2]=200 第30-40s: 300次 → seg[3]=300 第40-50s: 50次 → seg[4]=50 第50-60s: 10次 → seg[5]=10 ← current 总热度 = 100+150+200+300+50+10 = 810 → HIGH(≥500) → TTL=60+120=180s 10s后 rotate: current→seg[0], seg[0]归零 第60-70s: 5次 → seg[0]=5 总热度 = 5+150+200+300+50+10 = 715 ← 开始降 再过50s(第120s), seg[3]的300被rotate出去: 总热度 ≈ 50 → NONE → 回到基准TTL 60s
自然衰减： 不需要额外定时器。rotate 时自动丢弃最老那格数据，热度随时间自然降。
每个 key 24 字节（6×4B），100 万个 key = 24MB。
只延长不缩短 — TTL 到期后重新缓存，自然会用新（更短）的 TTL。

全链路串联

一次"刷 Feed → 点赞 → 关注"的完整旅程

三个系统不是孤立的。让用户 Alice 走一遍完整流程，看看各层如何协作。

场景设定
角色：Alice（userId=42）刚打开 App，刷到 Bob（userId=99）的帖子（postId=888）。
目标：Alice 浏览 Feed → 给帖子 888 点赞 → 关注 Bob → 看到粉丝数 +1。
时间线：这个过程跨越了 3 个系统、Redis 的 15+ 个 key、MySQL 的 3 张表、Kafka 的 2 个 topic。
阶段一：刷 Feed — Alice 看到帖子 888
Alice 打开 App → GET /api/v1/knowpost/feed?page=1&size=20 │ ├─ [Feed 流] Caffeine L0 miss → Redis L1 assembleFromCache │ LRANGE feed:public:ids:20:{hourSlot}:1 → ["888","777",...] │ MGET feed:item:888 feed:item:777 ... → 20条JSON │ ├─ [计数系统] 对每个帖子调用 counterService.getCounts() + isLiked() │ GET cnt:v1:knowpost:888 → byte[4..7]=42 (likeCount) │ GETBIT bm:like:knowpost:888:0 42 → 0 (Alice还没点赞) │ GET cnt:v1:knowpost:777 → byte[4..7]=15 │ GETBIT bm:like:knowpost:777:0 42 → 1 (Alice已经点过了) │ ← 20个帖子=20次GET+20次GETBIT，全部毫秒级 │ └─ enrich() 叠加 liked/faved → 返回给Alice Alice 看到帖子888：点赞数42，自己还没点赞（❤️空心）
关键：Feed 缓存只存标题/描述/图片。计数通过 counterService 实时查 SDS，点赞状态通过 GETBIT 实时查位图。缓存和状态分离——不同用户看到不同的 liked/faved，但共享同一份帖子元数据。
阶段二：点赞 — Alice 给帖子 888 点赞
Alice 点击 ❤️ → POST /api/v1/action/like {"entityType":"knowpost","entityId":"888"} │ ├─ [计数系统] CounterServiceImpl.toggle() │ chunk=42/32768=0, bit=42%32768=42 │ EVAL TOGGLE_LUA bm:like:knowpost:888:0 42 "add" │ GETBIT → 0 (之前没点过) → SETBIT 1 → return 1 (changed!) │ ├─ [计数] changed=1 → 两条消息同时发出： │ ① Kafka → CounterEvent(knowpost,888,like,idx=1,uid=42,delta=+1) │ ② Spring Event → FeedCacheInvalidationListener 即时更新Feed缓存 │ ├─ [Feed流] FeedCacheInvalidationListener.onCounterChanged() │ ① 加载 knowpost 888 的 creatorId → 发现作者是Bob(99) │ ② userCounterService.incrementLikesReceived(99, +1) → ucnt:99 likes+1 │ ③ SMEMBERS feed:public:index:888:{hourSlot} → 找到所有包含888的页面key │ ④ 对每个页面: 更新Caffeine本地缓存(保留liked) + 更新Redis页面JSON(清除liked, 保留原TTL) │ ← Alice立刻看到自己的 ❤️ 变红，赞数变 43 │ ├─ [Kafka异步] CounterAggregationConsumer.onMessage() │ HINCRBY agg:v1:knowpost:888 "1" 1 → 聚合桶 +1 → ack │ └─ [1秒后] flush() 定时任务 INCR_FIELD_LUA: cnt:v1:knowpost:888 的 byte[4..7] 从42→43 DECR_FIELD_LUA: agg:v1:knowpost:888 field"1" -1 → 扣减到0 → HDEL
两条路径的时序：Spring Event 路径 → 毫秒级（同步），Alice 点赞后立刻看到变化。
Kafka 路径 → 最长 1 秒延迟（异步），SDS 在 1 秒内更新。这个窗口内如果另一个用户刷 Feed 看到计数还是 42，但 Alice 自己看到的是 43（因为她的 liked 状态来自 BITMAP，计数来自 Spring Event 即时更新）。
阶段三：关注 — Alice 关注 Bob
Alice 点击关注 → POST /api/v1/relation/follow?toUserId=99 │ ├─ [用户关系] RelationServiceImpl.follow(42, 99) │ ① TOKEN_BUCKET_LUA rl:follow:42 → 有令牌 → 通过 │ ② @Transactional 开始 │ ③ INSERT INTO following (id=random, from=42, to=99, status=1) │ ④ INSERT INTO outbox (id=random, type="FollowCreated", payload={from:42,to:99,id:...}) │ ⑤ 事务提交 → MySQL binlog 写入两条 INSERT │ ├─ [Canal桥接] CanalKafkaBridge │ getWithoutAck → binlog里看到outbox的INSERT → 提取payload │ → kafka.send("canal-outbox", {table:"outbox",data:[{payload:"{...}"}]}) │ → ack(batchId) │ ├─ [Kafka消费] CanalOutboxConsumer → RelationEventProcessor │ ① SETNX dedup:rel:FollowCreated:42:99:{id} = "1", TTL=10min │ ② INSERT INTO follower (id, to=99, from=42, status=1) ← 投影表异步更新 │ ③ ZADD uf:flws:42 {now} 99 + EXPIRE 2h ← Alice的关注列表缓存 │ ④ ZADD uf:fans:99 {now} 42 + EXPIRE 2h ← Bob的粉丝列表缓存 │ ├─ [计数联动] UserCounterService │ ⑤ INCR_FIELD_LUA ucnt:42 → followings +1 (Alice关注数+1) │ ⑥ INCR_FIELD_LUA ucnt:99 → followers +1 (Bob粉丝数+1) │ └─ [自愈] 最多300秒后，/counter 端点采样校验 SELECT COUNT(*) WHERE from=42 AND status=1 → 对比 ucnt:42 followings字段 不一致 → rebuildAllCounters(42)
延迟分析：Alice 点击关注 → 事务提交（~5ms）→ Canal 拉取 binlog（~100ms）→ Kafka 投递（~50ms）→ 消费者处理（~10ms）→ follower 表 + ZSet + 计数器全部更新。端到端 200ms 以内，对用户体验无感知。

3分钟版本结构：
① 一句话概览（15s）：我用一个社交内容平台的项目，做了三件事——计数、关系、Feed 缓存。它们通过 Kafka 和 binlog 串在一起。
② 讲最难的（1min）：计数系统——两层模型（Bitmap+SDS），写路径走 Lua→Kafka→聚合桶→SDS，读路径 SDS 一次 GET。重建用三层保护。
③ 讲最不一样的（1min）：关系系统不用分布式事务，用 Outbox+Canal 保证事件不丢。和计数系统的区别：为什么一个用 Canal 一个不用。
④ 收尾（45s）：Feed 多级缓存——片段缓存+小时槽+反向索引+hotkey。三个系统通过一次"刷Feed→点赞→关注"串起来。

关键代码 — 逐行注释版

面试官问"能写出来吗"的时候，这些片段就是你的武器。

1. TOGGLE_LUA — 原子点赞翻转（Redis Lua）
-- KEYS[1] = bm:like:knowpost:123:0 (位图key) -- ARGV[1] = 42 (bit偏移) -- ARGV[2] = "add" 或 "remove" (操作类型) local bmKey = KEYS[1] local offset = tonumber(ARGV[1]) local op = ARGV[2] -- ① 先读当前位状态（关键！不能跳过这一步） local prev = redis.call('GETBIT', bmKey, offset) if op == 'add' then -- ② 如果已经是1 → 幂等返回0（不重复产生计数事件） if prev == 1 then return 0 end -- ③ 确实是0→1的转换 → 翻转并返回changed redis.call('SETBIT', bmKey, offset, 1) return 1 elseif op == 'remove' then -- ②' 同样：如果已经是0 → 幂等返回0 if prev == 0 then return 0 end -- ③' 1→0的转换 → 翻转并返回changed redis.call('SETBIT', bmKey, offset, 0) return 1 end return -1 -- 非法操作类型（不会发生）
为什么不能直接 SETBIT？ SETBIT 返回旧值，但两个并发 SETBIT 都返回不同的旧值（A返回0，B返回1），调用方都认为状态改变了。Lua 里先 GETBIT 判断再 SETBIT，在 Redis 单线程内原子完成，中间没有其他命令插入。
2. INCR_FIELD_LUA — 原子字段增量（Redis Lua）
-- KEYS[1] = cnt:v1:knowpost:123 (SDS key) -- ARGV[1] = 5 (schema长度/字段数) -- ARGV[2] = 4 (每字段字节数) -- ARGV[3] = 1 (字段索引, 0-based for "like") -- ARGV[4] = 1 (delta: +1 或 -1) -- 大端int32读取：从byte[off+1..off+4]拼接成整数 local function read32be(s, off) local b = {string.byte(s, off+1, off+4)} -- Lua的string从1开始! local n = 0 for i=1,4 do n = n * 256 + b[i] end -- 大端：高位在前 return n end -- 大端int32写入：整数拆成4个字节 local function write32be(n) local t = {} for i=4,1,-1 do t[i] = n % 256; n = math.floor(n/256) end return string.char(unpack(t)) -- 从高位到低位拼成字符串 end local cntKey = KEYS[1] local schemaLen = tonumber(ARGV[1]) local fieldSize = tonumber(ARGV[2]) local idx = tonumber(ARGV[3]) local delta = tonumber(ARGV[4]) local cnt = redis.call('GET', cntKey) -- ① SDS不存在 → 全零初始化（关键：保证系统冷启动可用） if not cnt then cnt = string.rep(string.char(0), schemaLen * fieldSize) end local off = idx * fieldSize local v = read32be(cnt, off) + delta -- ② 不能为负（防止取消点赞把计数拉到负值） if v < 0 then v = 0 end local seg = write32be(v) -- ③ 替换目标字段的4字节，其他字段不变（就地编辑） cnt = string.sub(cnt, 1, off) .. seg .. string.sub(cnt, off+fieldSize+1) redis.call('SET', cntKey, cnt) return 1
全部在 Redis 内原子执行。 GET→解码→+delta→编码→SET 之间不会有其他命令。但注意两个 Lua 脚本之间的原子性不保证——flush() 和 rebuild() 同时操作同一个 SDS key 时，各脚本独立原子，执行顺序取决于 Redis 事件循环。
3. Single-Flight 模式 — 防缓存击穿（Java）
// KnowPostFeedServiceImpl.java — getPublicFeed() // ConcurrentHashMap: key=idsKey(Redis级别), value=锁对象 private final ConcurrentHashMap<String, Object> singleFlight = new ConcurrentHashMap<>(); // ① computeIfAbsent: 原子地"有则返回，无则创建" // 如果 idsKey 已有锁对象 → 返回该锁 → 和其他线程共享同一把锁 // 如果 idsKey 没有锁 → 创建 new Object() → 只有这个线程拿到自己创建的锁 Object lock = singleFlight.computeIfAbsent(idsKey, k -> new Object()); synchronized (lock) { // ② Double Check: 在等锁期间，前一个线程可能已经回填了缓存 // 避免重复回源数据库 FeedPageResponse again = assembleFromCache(idsKey, ...); if (again != null) { feedPublicCache.put(localPageKey, again); singleFlight.remove(idsKey); // ← 释放锁对象，防止内存泄漏 return again; } // ③ 数据库回源（只有持锁线程执行） List<KnowPostFeedRow> rows = mapper.listFeedPublic(size + 1, offset); // ④ 回填缓存（其他等锁线程受益） writeCaches(...); feedPublicCache.put(localPageKey, respForCache); singleFlight.remove(idsKey); // ← 必须释放！否则所有后续请求永远等锁 return enrich(respForCache, currentUserId); } // ⚠ 风险：如果第③步抛异常，remove() 不执行 → 锁泄漏 → 死锁 // 生产应改为 try { ... } finally { singleFlight.remove(idsKey); }
为什么用 idsKey 而不是 localPageKey？ idsKey 是 Redis 层面的全局标识（跨应用实例），localPageKey 是应用本地的概念（含版本号）。用 idsKey 保证不同应用实例对同一页的请求共享同一把锁。
4. TOKEN_BUCKET_LUA — 令牌桶限流（Redis Lua）
-- KEYS[1] = rl:follow:42 (限流key，用Redis Hash存储) -- ARGV[1] = 100 (capacity，桶容量) -- ARGV[2] = 1 (rate，每秒生成令牌数) local key = KEYS[1] local capacity = tonumber(ARGV[1]) local rate = tonumber(ARGV[2]) -- ① 用 TIME 命令获取单调秒数（比服务器时钟可靠） local now = redis.call('TIME')[1] local last = redis.call('HGET', key, 'last') local tokens = redis.call('HGET', key, 'tokens') -- ② 首次请求：初始化满桶 if not last then last = now; tokens = capacity end -- ③ 计算流逝时间内补充的令牌数（匀速补充） local elapsed = tonumber(now) - tonumber(last) local add = elapsed * rate tokens = math.min(capacity, tonumber(tokens) + add) -- ④ 令牌不够 → 拒绝 + 更新状态（保留当前令牌数） if tokens < 1 then redis.call('HSET', key, 'last', now) redis.call('HSET', key, 'tokens', tokens) return 0 end -- ⑤ 消耗1个令牌 → 更新桶状态 → 设置60s过期（idle清理） tokens = tokens - 1 redis.call('HSET', key, 'last', now) redis.call('HSET', key, 'tokens', tokens) redis.call('PEXPIRE', key, 60000) return 1
100 capacity + 1/s rate 的含义： 用户可以 1 秒内 burst 关注 100 人（导入书签等），之后需要等 1 秒才能再关注。速率 1/s 是正常限流（每秒只能关注 1 人），容量 100 是 burst 容忍。
5. writeCaches — 写入三级缓存（Java）
// KnowPostFeedServiceImpl.java private void writeCaches(String pageKey, String idsKey, String hasMoreKey, int size, List<KnowPostFeedRow> rows, List<FeedItemResponse> items, boolean hasMore, Duration frTtl) { // ① 提取所有帖子ID → LPUSH 到 Redis List → 设TTL List<String> idVals = new ArrayList<>(); for (KnowPostFeedRow r : rows) idVals.add(String.valueOf(r.getId())); redis.opsForList().leftPushAll(idsKey, idVals); redis.expire(idsKey, frTtl); // 60-90s (含随机抖动) // ② hasMore 软缓存（短TTL，高浮动） if (idVals.size() == size && hasMore) { redis.opsForValue().set(hasMoreKey, "1", Duration.ofSeconds(10 + ThreadLocalRandom.current().nextInt(11))); } // ③ 页面键集合索引（用于批量失效与清理） redis.opsForSet().add("feed:public:pages", pageKey); for (FeedItemResponse it : items) { // ④ 反向索引：按小时槽记录"这个帖子出现在哪些页面上" long hourSlot = System.currentTimeMillis() / 3600000L; String idxKey = "feed:public:index:" + it.id() + ":" + hourSlot; redis.opsForSet().add(idxKey, pageKey); redis.expire(idxKey, frTtl); // ⑤ 条目缓存：每个帖子单独一个key，存完整JSON（无liked/faved） String itemKey = "feed:item:" + it.id(); String itemJson = objectMapper.writeValueAsString(it); redis.opsForValue().set(itemKey, itemJson, frTtl); } }
为什么每个 item 单独一个 key？ 当帖子被点赞时，反向索引定位到受影响的页面，可以精准更新单个 item key 中的计数。如果整页存一个 JSON，就得把整页读出来→改一个数字→写回去——修改成本高且耗时。
6. RelationEventProcessor — Outbox 事件处理（Java）
// RelationEventProcessor.java public void process(RelationEvent evt) { // ① 去重key: dedup:rel:{type}:{from}:{to}:{id} // 包含 following.id 保证：取关后重关注不会误判为重复 String dk = "dedup:rel:" + evt.type() + ":" + evt.fromUserId() + ":" + evt.toUserId() + ":" + (evt.id() == null ? "0" : String.valueOf(evt.id())); // ② SETNX: 幂等去重。已存在→跳过，首次→处理 Boolean first = redis.opsForValue() .setIfAbsent(dk, "1", Duration.ofMinutes(10)); if (first == null || !first) return; // 重复事件，跳过 if ("FollowCreated".equals(evt.type())) { // ③ 异步落库：投影表写入（字段反转） mapper.insertFollower(evt.id(), evt.toUserId(), evt.fromUserId(), 1); long now = System.currentTimeMillis(); // ④ ZSet 缓存：关注列表 + 粉丝列表双向更新 redis.opsForZSet().add("uf:flws:" + evt.fromUserId(), String.valueOf(evt.toUserId()), now); redis.opsForZSet().add("uf:fans:" + evt.toUserId(), String.valueOf(evt.fromUserId()), now); redis.expire("uf:flws:" + evt.fromUserId(), Duration.ofHours(2)); redis.expire("uf:fans:" + evt.toUserId(), Duration.ofHours(2)); // ⑤ 用户维度计数器：原子增量（Lua脚本） userCounterService.incrementFollowings(evt.fromUserId(), 1); userCounterService.incrementFollowers(evt.toUserId(), 1); } else if ("FollowCanceled".equals(evt.type())) { // 取关：软删除 + 移除ZSet + 计数器-1 mapper.cancelFollower(evt.toUserId(), evt.fromUserId()); redis.opsForZSet().remove("uf:flws:" + evt.fromUserId(), String.valueOf(evt.toUserId())); redis.opsForZSet().remove("uf:fans:" + evt.toUserId(), String.valueOf(evt.fromUserId())); userCounterService.incrementFollowings(evt.fromUserId(), -1); userCounterService.incrementFollowers(evt.toUserId(), -1); } }
注意： 如果第二步 ZSet 写失败（Redis OOM），follower 表已经写成功了 → 数据不一致。
当前代码没有显式的补偿机制——依赖 300s 采样校验和 rebuildAllCounters() 被动修复。生产环境应加 ZSet 写入的异常处理或 DLQ。
